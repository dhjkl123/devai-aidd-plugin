# 플러그인이 끼어드는 시점과 기능 안내

이 플러그인은 opencode가 작업하는 동안 **7개 시점**에서 끼어들어 git 관련 사고를 막고, 필요한 질문을 사용자에게 띄워줍니다. 시점마다 어떤 기능이 켜지는지 표로 정리했습니다.

---

## 한눈에 보는 7개 시점

| 시점 | 코드 식별자 (hook key) | 구현 파일 | 언제 동작하나요 | 사용자가 체감하는 효과 |
|---|---|---|---|---|
| 도구 등록 자리 | `tool` | `src/index.js:382` | 항상 (자리만 유지) | (현재 비어 있음) |
| 슬래시 명령 직전 | `command.execute.before` | `src/hooks/command-execute-before.js` | 사용자가 `/명령어`를 보낼 때 | 작업 시작 전 git 상태 점검하고 필요한 질문을 묶어서 띄움 |
| AI가 작업 시작 직전 | `tool.execute.before` | `src/hooks/tool-execute-before.js` | AI가 파일을 고치거나 명령을 돌리려 할 때 | 위험한 작업을 막고, 답변이 안 끝난 질문이 있으면 그쪽으로 안내 |
| AI 작업 끝난 직후 | `tool.execute.after` | `src/hooks/tool-execute-after.js` | 한 가지 작업을 마쳤을 때 | 결과를 정리해서 사용자에게 보여주고, 다음 단계 질문이 필요하면 띄움 |
| 사용자 답변 받은 직후 | `permission.asked` | `src/hooks/permission-asked.js` | 승인/거절/복구 버튼을 눌렀을 때 | 누른 버튼대로 실제 git 작업을 실행하거나 다음 단계로 진행 |
| 파일 편집 직후 | `file.edited` | `src/hooks/file-edited.js` | AI가 파일을 만들거나 수정했을 때 | 어떤 파일이 바뀌었는지 조용히 기록 (나중에 커밋 질문에 사용) |
| 종합 알림 라우터 | `event` | `src/hooks/native-event.js` | opencode가 보내는 모든 종류의 알림 | 알림 종류를 구분해서 알맞은 처리로 넘김 |

## 1. 도구 등록 자리 — `tool`

| 항목 | 내용 |
|---|---|
| 제공 기능 | 없음 (자리만 유지) |
| 이유 | 이전 버전에서 쓰던 "시작 승인용 도구"를 제거한 후 외부 약속을 깨지 않으려고 빈 자리만 남겨둠 |

---

## 2. 슬래시 명령 실행 직전 — `command.execute.before`

사용자가 `/quick-spec`, `/dev-story` 같은 명령을 보낼 때 본격 실행 전에 git 환경을 점검하고 필요한 질문을 한꺼번에 띄웁니다.

| 기능 | 사용자가 체감하는 동작 |
|---|---|
| 워크플로우 명령 여부 판별 | 입력한 명령이 안내가 필요한 워크플로우 명령인지 확인. 일반 명령이면 그대로 통과 |
| 작업 폴더 상태 점검 | 이 폴더가 git 사용 가능한 상태인지 4가지로 판별 (아래 별도 표) |
| 시작 질문 한 번에 묶기 | 점검 결과에 따라 "git 시작 → 첫 저장(커밋) → 새 작업 공간(브랜치)"을 한꺼번에 묶어 띄움 |
| AI에게 안내문 자동 전달 | "지금부터 git 안전장치가 켜져 있다"는 안내를 AI에게 자동으로 알려서 임의로 git을 건드리지 못하게 함 |
| 이미 "건너뛰기"를 골랐을 때 | 이번 세션 동안에는 git 관련 질문/차단을 모두 끄고 워크플로우 본 작업만 진행 |
| 작업 기록 남기기 | "워크플로우 감지됨" 같은 기록을 감사 로그에 남김 |

### 작업 폴더 상태 점검에서 구분하는 4가지

| 점검 결과 | 의미 | 다음 동작 |
|---|---|---|
| 아직 git 시작 안 됨 | 이 폴더에 git이 한 번도 켜진 적이 없음 | "git을 시작하시겠어요?" 질문 띄움 |
| 시작은 했지만 저장 0번 | git은 켜졌지만 아직 한 번도 저장(커밋)한 적이 없음 | "기준이 되는 첫 저장을 만들까요?" 질문 띄움. 이 상태에서는 새 작업 공간(브랜치)을 만들지 못하게 함 (만들어도 망가지기 때문) |
| 정상 사용 가능 | git이 켜져 있고 저장 기록도 있음 | 워크플로우 규칙에 따라 새 작업 공간 생성 여부 결정 |
| 점검 자체 실패 | git 점검 명령이 어떤 이유로 안 됨 | 강제로 차단하지 않고 작업을 그대로 진행 (사용자 작업 흐름을 끊지 않기 위한 안전망) |

## 3. AI가 작업을 시작하기 직전 — `tool.execute.before` (가장 강한 안전장치)

AI가 파일을 고치거나 명령을 돌리려는 순간, 다음 규칙을 차례로 점검해서 위험하면 막습니다.

| 안전 규칙 | 어떤 상황을 막는지 | 사용자/AI가 받는 안내 |
|---|---|---|
| AI가 git 명령을 임의로 못 돌리게 막기 | "git 시작" 질문이 떠 있는데 AI가 몰래 git을 직접 돌리려 할 때 | "임의로 돌리지 말고 떠 있는 질문에 먼저 답하세요" |
| 답변 대기 중에는 다른 작업 금지 | 사용자에게 띄운 승인 질문이 답변 안 된 상태에서 AI가 다른 작업을 하려 할 때 | "지금 떠 있는 질문에 먼저 답해야 합니다" + 정확히 어떤 헤더/옵션으로 질문이 떠 있는지 안내 |
| 묶음 질문 대기 중에도 다른 작업 금지 | 시작 단계의 묶음 질문이 아직 안 띄워졌는데 AI가 다른 작업을 하려 할 때 | "묶음 질문을 먼저 띄워서 답을 받으세요" |
| 질문 헤더 함부로 바꾸지 못하게 함 | AI가 임의로 질문 문구를 바꿔서 띄우려 할 때 | "정확히 이 문구와 이 선택지로 다시 띄우세요" |
| 작업 공간 없이 파일 수정 금지 | 워크플로우는 시작됐지만 전용 작업 공간(브랜치)으로 옮기지 않았는데 파일을 고치려 할 때 | "먼저 전용 작업 공간으로 이동/생성하세요" |
| 진행 단계 표시 갱신 | 위 규칙을 모두 통과했을 때 | 워크플로우 진행 단계를 "진행 중"으로 업데이트 |

### 도구 종류별 처리

| 종류 | 어떤 작업인지 | 코드 식별자 (도구 이름) | 안전장치 동작 |
|---|---|---|---|
| 파일을 실제로 바꾸는 작업 | 파일 편집/생성/덮어쓰기 | `edit`, `write`, `patch`, `multiedit` | 작업 공간(브랜치) 안에 있어야만 허용 |
| 읽기만 하는 안전한 작업 | 파일 읽기, 검색, 웹 조회 등 | `read`, `glob`, `grep`, `list`, `lsp`, `webfetch`, `websearch`, `codesearch`, `skill`, `todoread` | 막지 않고 통과 |
| 사용자에게 질문 띄우는 작업 | 승인/선택지 질문 | `question` | 통과시키되, 문구가 정확한지만 확인 |
| 마무리 신호 | AI가 작업 완료를 알리는 도구 | `finish` | 변경된 파일 점검 → 커밋/푸시 승인 질문 띄우기 (`tool.execute.after`에서 처리) |

## 4. AI 작업이 끝난 직후 — `tool.execute.after`

| 분기 | 발동 시점 | 사용자가 체감하는 결과 |
|---|---|---|
| 시작 묶음 질문 답변 처리 | 사용자가 "git 시작 / 첫 저장 / 새 작업 공간" 묶음 질문에 답한 직후 | 답변대로 git 작업을 실제로 순서대로 실행하고, 무엇이 됐고 무엇이 건너뛰어졌는지 요약을 AI 응답에 자동으로 붙임 |
| 실행 중 오류 발생 시 복구 안내 | 위 실행 도중 git 작업이 실패했을 때 | "다시 시도 / 자동화 끄고 계속 / 직접 처리 / 그만두기" 같은 복구 선택지를 띄움 |
| 워크플로우 마무리 신호 처리 | AI가 작업 마무리 신호를 보냈을 때 | 바뀐 파일이 있으면 "이걸로 저장(커밋)할까요?", 필요하면 "원격에 올릴까요?(푸시)" 같은 마무리 질문을 띄움 |
| 진행 단계 표시 갱신 | 그 외 일반 작업 직후 | 파일을 고친 작업이면 "수정 중", 아니면 "진행 중" 단계로 표시 |

## 5. 사용자가 승인 창에 답한 순간 — `permission.asked`

승인 창(또는 복구 선택 창)에서 사용자가 어떤 버튼을 눌렀느냐에 따라 다음 행동이 달라집니다.

| 사용자가 누른 선택지 | 플러그인 동작 |
|---|---|
| 승인 (예: "Git 시작", "예", "진행") | 실제 git 작업을 실행. 성공하면 다음 단계 진행, 실패하면 자동으로 복구 창을 띄움 |
| 거절 (예: "취소", "아니오") | 작업 중단. 다시 시도가 가능한 작업이면 복구 선택 창을 띄움 |
| 무시하고 계속 (예: "건너뛰기") | 이번 git 작업을 건너뛰고 워크플로우만 계속 진행. 이후 같은 종류의 git 자동화는 이번 세션 동안 꺼둠 |
| 복구 선택 (재시도/자동화 끄기/직접 처리/그만두기) | 사용자가 고른 방식대로 다음 행동 결정 |
| 매칭되는 대기 항목이 없음 | 아무 일도 하지 않음 (관련 없는 응답은 안전하게 무시) |

## 6. 파일 편집 직후 — `file.edited` (변경 파일 목록 기록)

| 기능 | 사용자가 체감하는 효과 |
|---|---|
| 워크플로우 세션인지 확인 | 일반 대화(슬래시 명령으로 시작되지 않은 세션)에서는 아무것도 안 함 |
| 편집 파일 경로 정리 | 절대 경로를 작업 폴더 기준 상대 경로로 변환해 깔끔하게 정리 |
| 중복 제거 | 같은 파일이 여러 번 편집돼도 목록에는 한 번만 기록 |
| 마무리 시 활용 | 워크플로우가 끝날 때 "이 파일들을 저장(커밋)할까요?" 같은 질문에 자동으로 활용 |

## 7. opencode가 보내는 모든 알림의 중앙 라우터 — `event`

opencode 본체가 보내는 알림을 종류별로 구분해 알맞은 처리로 넘깁니다.

| 알림 종류 | 코드 식별자 (event.type) | 의미 | 플러그인 처리 |
|---|---|---|---|
| 슬래시 명령 실행됨 | `command.executed` | 사용자가 `/명령어`를 보냄 | 위 2번과 같은 처리(작업 폴더 점검 + 시작 질문 띄우기)를 그대로 실행 |
| 질문 띄워짐 | `question.asked` | opencode가 사용자에게 질문 창을 보여줌 | 어떤 종류의 질문인지(시작 묶음 / 복구 / 일반 승인) 분류해서 대기 상태로 기록 |
| 질문 답변됨 | `question.replied` | 사용자가 답을 입력함 | 답 내용에 따라 git 작업 실행 / 복구 진행 / 다음 단계로 분기 |
| 질문 닫힘 | `question.rejected` | 사용자가 X 버튼으로 질문 창을 닫음 | 일반 승인은 "건너뛰기"로 처리, 복구 질문은 대기만 풀고 그대로 둠(다시 물을 수 있게) |
| 세션 종료됨 | `session.deleted` | 대화 세션이 끝남 | 이 세션에서 기록한 모든 작업 상태를 깨끗이 정리 |
| 그 외 알림 | (그 외 타입) | 워크플로우와 무관한 알림 | 그대로 무시 |

### 7-1. 어떤 질문인지 판별하는 우선순위

| 순위 | 조건 | 어떤 질문으로 인식 |
|---|---|---|
| 1 | 시작 묶음 질문 진행 중 | 시작 묶음 질문 |
| 2 | 복구 절차 진행 중 | 복구 선택 질문 |
| 3 | 일반 git 승인이 떠 있음 | 단일 승인 질문 |
| 4 | 위 모두 해당 없음 | 무시 (워크플로우와 무관한 질문) |

### 7-2. 답변 단어 인식

사용자가 영어/한국어 등 다양한 표현으로 답할 수 있어, 같은 의미의 표현을 미리 정리해 두고 인식합니다.

| 답변 의미 | 인식하는 표현 (대소문자 무시) |
|---|---|
| 승인 | "initialize", "initialize git", "yes", "approve", "approved", "allow", "accept", "proceed", "continue", "ok" |
| 거절 | "cancel", "no", "deny", "reject", "block", "stop" |
| 건너뛰기 | "ignore", "ignore-and-continue", "skip" |
| 복구 선택지 | "retry"(다시 시도), "continue-without-automation"(자동화 끄고 계속), "manual"(직접 처리), "abandon"(그만두기) 등 |

## 부록 — 시점들이 공통으로 쓰는 도우미 (`pluginContext`)

7개 시점이 함께 사용하는 내부 도우미들입니다. 모두 `src/index.js`의 `pluginContext`로 주입됩니다.

| 도우미 | 코드 식별자 | 하는 일 |
|---|---|---|
| git 명령 실행기 | `pluginContext.gitRunner` | git 명령을 안전하게 돌리고 결과를 받아옴 |
| git 작업 실행기 | `pluginContext.gitActionRunner` | "시작", "저장", "작업 공간 생성", "원격에 올리기" 같은 의미 단위로 묶어 실행 |
| 워크플로우 규칙 해석기 | `pluginContext.resolvePolicy` | 현재 명령에 어떤 git 규칙(예: 브랜치 필수 여부)이 붙는지 해석 |
| 변경 파일 조회기 | `pluginContext.listChangedFiles` | git이 추적하는 변경 파일 목록을 가져옴 |
| 단일 승인 질문 띄우기 | `pluginContext.requestApproval` | 한 가지 git 작업에 대한 승인 질문을 화면에 띄움 |
| 묶음 질문 띄우기 | `pluginContext.requestStartupChainApproval` | 시작 단계의 묶음 질문을 한꺼번에 띄움 |
| 복구 선택 질문 띄우기 | `pluginContext.requestRecoveryDecision` | 작업 실패 시 복구 옵션 질문을 띄움 |
| 진단 로그 | `pluginContext.debug` | 설정에서 디버그가 켜져 있을 때만 진단 기록을 남김 |

| 승인 | git 작업(시작/저장/작업 공간 생성/원격 올리기)을 실제로 하기 전 사용자에게 받는 OK |
| 복구 | 승인된 작업이 실패했을 때 사용자에게 보여주는 다음 행동 선택지 |
| 시작 묶음 질문 | 워크플로우 시작 전에 git 시작/첫 저장/작업 공간 생성을 한꺼번에 묶어 물어보는 질문 세트 |
| 첫 저장(베이스라인 커밋) | 빈 저장소에 처음 만드는 기준 저장. 이게 없으면 새 작업 공간을 만들 수 없음 |
| 작업 공간(브랜치) | 본 코드와 분리해서 안전하게 변경 작업을 할 수 있는 별도 공간 |
| 원격에 올리기(푸시) | 내 컴퓨터의 저장 내용을 팀 공용 저장소로 보내는 작업 |
| 세션 | 사용자가 opencode와 한 번 시작해서 종료할 때까지의 대화 단위 |
| 마무리 신호 | AI가 작업이 끝났음을 알리는 내부 신호 |
| 안전 작업 vs 위험 작업 | 읽기만 하는 작업은 통과시키고, 파일을 실제로 바꾸는 작업은 안전장치를 통과해야 진행됨 |
