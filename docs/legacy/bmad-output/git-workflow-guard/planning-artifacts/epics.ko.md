# opencode-aidd-plugin - 에픽 분해

## 개요

이 문서는 opencode-aidd-plugin의 전체 에픽 및 스토리 분해를 제공합니다. PRD의 요구사항, 아키텍처의 소스 인식 기술 제약조건, 현재 구현 기준선을 구현 가능한 스토리로 분해합니다.

## 요구사항 목록

### 기능 요구사항

FR1: 시스템은 어떤 BMAD 워크플로우 명령이 시작되었는지 감지해야 한다.

FR2: 시스템은 감지된 워크플로우 컨텍스트에 맞는 Git 정책을 해석하고 적용해야 한다.

FR3: 시스템은 활성 워크플로우 컨텍스트에 적합한 브랜치 전략을 선택하거나 제안해야 한다.

FR4: 시스템은 워크플로우의 시작, 진행 중, 완료 단계를 구분하여 각 단계에서 다른 Git 작업을 적용할 수 있어야 한다.

FR5: 시스템은 워크플로우 시작 시 구성된 규칙에 맞는 브랜치를 생성하거나 전환해야 한다.

FR6: 시스템은 구성된 명명 규칙으로부터 후보 브랜치 이름을 생성하거나 제안해야 한다.

FR7: 시스템은 워크플로우 종료 시 워크플로우 출력 아티팩트로부터 커밋을 준비하거나 생성해야 한다.

FR8: 시스템은 원격 저장소가 구성된 경우 커밋 후 푸시를 제안해야 한다.

FR9: 시스템은 작업 디렉토리가 아직 로컬 Git 저장소가 아닌 경우 저장소 초기화를 제안해야 한다.

FR10: 시스템은 모든 Git 작업 실행 전에 사용자 승인을 요청해야 한다.

FR11: 시스템은 승인 전에 제안된 각 Git 작업의 의도와 예상 효과를 설명해야 한다.

FR12: 시스템은 제안된 각 Git 작업에 대해 사용자가 수락, 거부 또는 무시 후 계속하기를 선택할 수 있어야 한다.

FR13: 시스템은 Git 작업이 거부되거나 무시된 경우에도 워크플로우가 계속될 수 있어야 한다.

FR14: 관리자는 JSON 기반 구성을 통해 브랜치 명명 규칙을 정의하거나 변경할 수 있어야 한다.

FR15: 관리자는 JSON 기반 구성을 통해 워크플로우별 Git 정책을 정의하거나 변경할 수 있어야 한다.

FR16: 시스템은 프로젝트 수준 및 전역 구성을 모두 로드하고 결정론적 우선순위를 적용해야 한다.

FR17: 시스템은 레거시 구성 형식을 읽고 호환성 동작을 유지해야 한다.

FR18: 관리자는 팀별 Git 정책에 맞게 자동화 동작을 조정할 수 있어야 한다.

FR19: 시스템은 브랜치 충돌, 커밋 실패, 푸시 실패 및 저장소 상태 불일치에 대해 사용자에게 명확히 알려야 한다.

FR20: 시스템은 예외의 원인을 설명하고 사용 가능한 복구 옵션을 제시해야 한다.

FR21: 시스템은 재시도, 건너뛰기 또는 수동 해결 후 계속하기와 같은 복구 경로를 제공해야 한다.

FR22: 시스템은 Git 자동화 실패가 전체 BMAD 워크플로우 실패로 즉시 이어지는 것을 방지해야 한다.

FR23: 시스템은 코드, 기술 문서 및 기획 아티팩트를 Git 이력에 기록해야 한다.

FR24: 리뷰어는 Git 기록을 통해 아티팩트의 작성자와 변경 이력을 추적할 수 있어야 한다.

FR25: 리뷰어는 표준 Git 도구를 사용하여 특정 아티팩트의 변경 책임을 검사할 수 있어야 한다.

FR26: 시스템은 사용자 승인 결과와 실행된 Git 작업 결과를 추적 가능한 형태로 노출해야 한다.

FR27: 플러그인은 opencode/DevAI 런타임 플러그인 및 훅 시스템과의 통합으로 실행되어야 한다.

FR28: 플러그인은 세션 이벤트와 도구 실행 전/후 훅 흐름을 사용하여 Git 정책을 적용해야 한다.

FR29: 플러그인은 사용자 측 명령 변경 없이 기존 코어 BMAD 워크플로우 명령과의 호환성을 유지해야 한다.

FR30: 플러그인은 Node.js 기반 opencode/DevAI 런타임에서 공식적으로 지원되는 동작을 제공해야 한다.

### 비기능 요구사항

NFR1: 워크플로우 컨텍스트 감지 및 Git 제안 로직은 사용자의 BMAD 흐름을 방해하는 불필요한 지연을 추가해서는 안 된다.

NFR2: 사용자 승인 후, 시스템은 Git 실행 피드백을 신속히 반환하고 지연 발생 시 AI 지연과 로컬 Git 처리 지연을 구분해야 한다.

NFR3: 명시적 사용자 승인 없이는 어떤 Git 작업도 실행될 수 없다.

NFR4: 로그 및 감사 기록은 현재 BMAD 실행과 관련된 파일 및 워크플로우 활동으로 범위가 제한되어야 한다.

NFR5: 로그는 민감한 정보를 저장하지 않고 추적에 필요한 최소한의 승인 및 실행 메타데이터만 보존해야 한다.

NFR6: 시스템은 Git 자동화를 제안하거나 실행할 때 의도하지 않은 파일이나 저장소를 대상으로 해서는 안 된다.

NFR7: Git 자동화가 실패하면 시스템은 워크플로우를 계속하거나 중단할 사용자의 선택을 보존해야 한다.

NFR8: 실패는 이해 가능한 원인과 복구 가능한 옵션과 함께 제시되어야 한다.

NFR9: 브랜치 충돌, 커밋 실패, 푸시 실패 및 저장소 상태 불일치는 일관되게 감지하고 보고해야 한다.

NFR10: 플러그인은 현재 Node.js 기반 opencode/DevAI 런타임에서 완전한 호환성을 유지해야 한다.

NFR11: 플러그인은 원격 저장소가 구성되지 않은 경우를 확실히 감지해야 한다.

NFR12: 플러그인은 로컬 Git 저장소가 초기화되지 않은 경우를 확실히 감지해야 한다.

NFR13: 플러그인은 관련 자동화를 제안하거나 실행하기 전에 Git 상태와 런타임 컨텍스트를 모두 확인해야 한다.

NFR14: 지원되지 않는 런타임이 공식 Node.js 런타임과 동일한 보증을 제공하는 것으로 표시되어서는 안 된다.

### 추가 요구사항

- 새 외부 스캐폴드를 채택하는 대신 기존 저장소 기준선을 시작 템플릿으로 사용한다; 구현은 브라운필드 런타임 계약을 유지해야 한다.
- `src/index.js`를 레거시 코어를 래핑하고 호스트 플러그인 시스템이 기대하는 런타임 훅 맵을 노출하는 부트스트랩 진입점으로 유지한다.
- 전역, 프로젝트, 레거시 구성 파일을 읽고, 결정론적 순서로 병합하며, 필요 시 레거시 미러 파일을 작성하는 호환성 브릿지를 유지한다.
- 워크플로우 명령 매핑, 브랜치 명명 패턴 지원, 명령 유형 매핑, 브랜치 필수 동작, 완료 모드 선택이 포함된 현재 정책 모델을 유지한다.
- 훅 진입점을 가볍게 유지하고, 각 훅에서 로직을 중복하는 대신 중앙 집중식 오케스트레이션/서비스를 통해 실질적인 동작을 라우팅한다.
- 외부 구조를 리팩토링하면서 레거시 코어 동작을 유지한다; 래퍼 핸들러는 회귀 커버리지에 반영된 대로 레거시 구현과 동작적으로 동등해야 한다.
- `permission.asked`, `file.edited`, HTTP 감사 포워딩을 위한 현재 스텁 통합 포인트를 구현하거나 공식화한다(해당 기능이 범위 내로 주장되는 경우).
- 감사 로깅을 최선 노력 및 비차단으로 유지하여 로깅 실패가 플러그인 흐름을 중단하지 않도록 한다.
- 클라이언트 로깅, 상대 또는 절대 경로 처리가 가능한 선택적 파일 로깅, 감사 기록에 대한 선택적 HTTP 포워딩을 지원한다.
- JSON/JSONC 구성과 인메모리/세션 상태를 사용하는 파일 기반 지속성 모델을 유지한다; MVP에는 데이터베이스를 도입하지 않는다.
- ESM Node.js 패키징과 `esbuild`, 릴리스 매니페스트, SHA-256 체크섬 생성 기반의 스크립트 기반 빌드/릴리스 흐름을 유지한다.
- 릴리스 패키징에는 빌드된 플러그인, 설치기, 템플릿, 매니페스트, 체크섬이 `latest` 및 버전별 릴리스 디렉토리 모두에 포함되어야 한다.
- 설치기 아티팩트 및 템플릿 구성 파일을 소스 코드와 함께 일급 결과물로 유지한다.
- 명령 프롬프트와 변경 도구 보호에 대해 래퍼 동작과 빌드된 아티팩트 동작을 레거시 플러그인과 비교하는 회귀 테스트 스위트를 유지한다.
- 문서 범위에는 설치 가이드, 구성 가이드, 브랜치 규칙 예제, 승인 흐름 설명, 레거시 호환성 동작이 포함되어야 한다.

## 에픽 목록

### 에픽 1: 워크플로우 인식 안전 시작
사용자가 올바른 런타임 컨텍스트, Git 준비 상태 검사, 브랜치 설정 결정이 이미 준비된 상태에서 BMAD 워크플로우를 시작할 수 있다.
**대상 FR:** FR1, FR2, FR3, FR4, FR5, FR6, FR9, FR27, FR28, FR30

### 에픽 2: 승인 기반 Git 실행 및 복구
사용자가 제안된 Git 작업을 검토, 승인, 거부 또는 건너뛸 수 있으며, 자동화가 충돌이나 실패를 만나도 안전하게 워크플로우를 계속할 수 있다.
**대상 FR:** FR10, FR11, FR12, FR13, FR19, FR20, FR21, FR22

### 에픽 3: 완료 및 추적 가능한 전달
사용자가 커밋 및 선택적 푸시 동작으로 워크플로우를 완료하면서 코드, 문서, 기획 아티팩트에 대한 완전한 Git 기반 추적성을 유지할 수 있다.
**대상 FR:** FR7, FR8, FR23, FR24, FR25, FR26

### 에픽 4: 정책 관리 및 호환성 운영
관리자가 정책 구성을 관리하고, 레거시 호환성을 유지하며, 신뢰할 수 있는 패키징, 릴리스, 런타임 호환성 제어를 통해 플러그인을 운영할 수 있다.
**대상 FR:** FR14, FR15, FR16, FR17, FR18, FR29

## 에픽 1: 워크플로우 인식 안전 시작

사용자가 올바른 런타임 컨텍스트, Git 준비 상태 검사, 브랜치 설정 결정이 이미 준비된 상태에서 BMAD 워크플로우를 시작할 수 있다.

### 스토리 1.1: 플러그인 부트스트랩을 통한 런타임 훅 등록

플러그인 운영자로서,
플러그인 부트스트랩이 예상되는 opencode/DevAI 훅을 등록하기를 원한다.
이를 통해 런타임이 플러그인을 로드할 때 워크플로우 인식 Git 가드 동작이 일관되게 활성화된다.

**인수 조건:**

**주어진 조건** 플러그인이 지원되는 Node.js opencode/DevAI 런타임에서 로드된 경우
**행동 시점** 부트스트랩 진입점이 실행될 때
**기대 결과** 명령, 도구, 권한, 파일 편집, 세션 이벤트에 대한 예상 훅 맵을 반환한다
**그리고** 부트스트랩은 명령 변경 없이 기존 레거시 코어 통합 계약을 유지한다.

**주어진 조건** 런타임이 지원되지 않거나 불완전한 환경에서 플러그인을 로드하는 경우
**행동 시점** 부트스트랩 의존성이 누락되거나 유효하지 않을 때
**기대 결과** 플러그인이 저장소 상태를 손상시키지 않는 제어된 방식으로 실패한다
**그리고** 지원되는 런타임 요구사항이 로그 또는 진단에 명시된다.

### 스토리 1.2: BMAD 워크플로우 명령 및 런타임 컨텍스트 감지

워크플로우 사용자로서,
시스템이 내가 시작한 BMAD 명령과 실행 단계를 인식하기를 원한다.
이를 통해 이후 Git 결정이 올바른 워크플로우 컨텍스트에 기반하게 된다.

**인수 조건:**

**주어진 조건** 런타임에서 실행 전에 명령이 수신된 경우
**행동 시점** 명령이 구성되거나 발견된 BMAD 워크플로우 명령과 일치할 때
**기대 결과** 시스템은 워크플로우 정체성을 분류하고 하류 정책 해석을 위해 명령 컨텍스트를 기록한다
**그리고** 감지는 사용자가 기존 BMAD 명령을 변경할 필요 없이 작동한다.

**주어진 조건** 워크플로우 시작 후 세션 또는 도구 이벤트가 발생한 경우
**행동 시점** 런타임이 이벤트를 처리할 때
**기대 결과** 시스템은 시작, 진행 중, 완료 단계를 구분하기에 충분한 컨텍스트를 유지한다
**그리고** 비워크플로우 명령은 관련 없는 런타임 동작을 차단하지 않고 무시된다.

### 스토리 1.3: 병합된 구성 로드 및 워크플로우 정책 해석

팀 관리자로서,
플러그인이 전역, 프로젝트, 레거시 호환 설정을 하나의 유효 정책으로 로드하기를 원한다.
이를 통해 수동 코드 변경 없이 워크플로우 동작이 팀 규칙과 일치하게 된다.

**인수 조건:**

**주어진 조건** 전역 및 프로젝트 구성 파일이 모두 존재하는 경우
**행동 시점** 런타임 구성이 로드될 때
**기대 결과** 플러그인은 결정론적 우선순위를 적용하고 정규화된 유효 구성을 생성한다
**그리고** 브랜치 기본값, 장기 브랜치 규칙, 워크플로우 정책 매핑이 하류 핸들러에서 사용 가능하다.

**주어진 조건** 레거시 구성 파일만 존재하는 경우
**행동 시점** 플러그인이 구성을 로드할 때
**기대 결과** 레거시 파일을 읽고 필요 시 브릿지 파일을 생성하여 호환성을 유지한다
**그리고** 사용 전 마이그레이션을 강제하지 않고 워크플로우가 계속될 수 있다.

### 스토리 1.4: 브랜치 전략 및 후보 브랜치 이름 계산

워크플로우 사용자로서,
시스템이 활성 워크플로우에 적합한 브랜치 동작을 계산하기를 원한다.
이를 통해 브랜치 생성 또는 전환이 구성된 명명 및 정책 규칙을 자동으로 따르게 된다.

**인수 조건:**

**주어진 조건** 워크플로우 명령이 식별되고 정책이 해석된 경우
**행동 시점** 플러그인이 해당 워크플로우의 브랜치 동작을 평가할 때
**기대 결과** 워크플로우에 브랜치가 필수인지, 선택인지, 불필요한지 결정한다
**그리고** 구성된 명령 유형, 티켓 컨텍스트, 폴백 값, 슬러그 규칙으로부터 후보 브랜치 이름을 계산한다.

**주어진 조건** 현재 브랜치가 장기 브랜치이거나 워크플로우 정책을 만족하지 않는 경우
**행동 시점** 브랜치 평가가 실행될 때
**기대 결과** 플러그인은 Git 상태를 조용히 변경하는 대신 브랜치 생성 또는 전환 제안을 준비한다
**그리고** 제안은 사용자 승인을 별도의 이후 단계로 유지한다.

### 스토리 1.5: 저장소 준비 상태 확인 및 초기화 제안

워크플로우 사용자로서,
자동화가 시작되기 전에 플러그인이 내 작업 디렉토리가 Git 준비 상태인지 확인하기를 원한다.
이를 통해 저장소 제약 조건을 완전히 인식한 상태에서 초기화를 선택하거나 안전하게 계속할 수 있다.

**인수 조건:**

**주어진 조건** 초기화되지 않은 Git 저장소인 디렉토리에서 워크플로우가 시작된 경우
**행동 시점** 준비 상태 검사가 실행될 때
**기대 결과** 플러그인이 누락된 저장소 상태를 감지하고 초기화 제안을 준비한다
**그리고** 사용자가 명시적으로 승인할 때까지 `git init` 작업이 실행되지 않는다.

**주어진 조건** 유효한 저장소에서 워크플로우가 시작된 경우
**행동 시점** 준비 상태 검사가 환경을 검사할 때
**기대 결과** 플러그인이 시작 자동화와 관련된 브랜치 및 원격 전제 조건을 감지한다
**그리고** BMAD 워크플로우에 불필요한 지연을 추가하지 않고 저장소 준비 상태를 보고한다.

## 에픽 2: 승인 기반 Git 실행 및 복구

사용자가 제안된 Git 작업을 검토, 승인, 거부 또는 건너뛸 수 있으며, 자동화가 충돌이나 실패를 만나도 안전하게 워크플로우를 계속할 수 있다.

### 스토리 2.1: 계획된 Git 작업에 대한 승인 요청 제시

워크플로우 사용자로서,
계획된 각 Git 작업이 승인 요청으로 제시되기를 원한다.
이를 통해 저장소 변경이 발생하기 전에 자동화 진행 여부를 결정할 수 있다.

**인수 조건:**

**주어진 조건** 플러그인이 브랜치 생성, 브랜치 전환, 커밋, 푸시, 초기화 등의 Git 작업을 계획한 경우
**행동 시점** 해당 작업에 대한 승인이 필요할 때
**기대 결과** 런타임이 실행 전에 승인 요청을 제시한다
**그리고** 요청은 현재 워크플로우 세션 및 작업 유형과 연결된다.

**주어진 조건** 워크플로우 전반에 걸쳐 여러 Git 작업이 발생할 수 있는 경우
**행동 시점** 승인 프롬프트가 생성될 때
**기대 결과** 각 프롬프트는 단일 계획된 작업으로 범위가 제한된다
**그리고** 시스템은 현재 승인 결과가 해결될 때까지 이후 작업의 실행을 방지한다.

### 스토리 2.2: 승인 프롬프트에서 의도 및 예상 영향 설명

워크플로우 사용자로서,
승인 프롬프트가 플러그인이 무엇을 하려는지와 그 이유를 설명하기를 원한다.
이를 통해 각 Git 작업에 대해 정보에 기반한 결정을 내릴 수 있다.

**인수 조건:**

**주어진 조건** 계획된 Git 작업에 대한 승인 프롬프트가 표시된 경우
**행동 시점** 프롬프트가 사용자에게 렌더링될 때
**기대 결과** 작업 의도, 예상 저장소 영향, 워크플로우 컨텍스트를 사람이 읽을 수 있는 용어로 설명한다
**그리고** 프롬프트 본문이나 메타데이터에 불필요한 민감 정보를 노출하지 않는다.

**주어진 조건** 작업이 브랜치 또는 워크플로우 정책 규칙에 기반한 경우
**행동 시점** 설명이 구성될 때
**기대 결과** 사용자가 작업이 제안된 이유를 이해할 수 있도록 충분한 정책 컨텍스트를 포함한다
**그리고** BMAD 워크플로우를 불필요하게 방해하지 않을 만큼 간결하게 유지된다.

### 스토리 2.3: 수락, 거부, 무시 후 계속하기 결과 지원

워크플로우 사용자로서,
제안된 각 Git 작업을 수락, 거부 또는 무시할 수 있기를 원한다.
이를 통해 워크플로우의 추진력을 잃지 않으면서 자동화에 대한 통제를 유지할 수 있다.

**인수 조건:**

**주어진 조건** 승인 요청이 활성 상태인 경우
**행동 시점** 사용자가 수락, 거부 또는 무시 후 계속하기를 선택할 때
**기대 결과** 플러그인은 현재 작업 및 세션 컨텍스트에 대해 선택된 결과를 기록한다
**그리고** 후속 동작은 선택된 결과를 일관되게 따른다.

**주어진 조건** 사용자가 Git 작업을 거부하거나 무시한 경우
**행동 시점** 플러그인이 승인을 처리할 때
**기대 결과** 해당 작업에 대해 저장소는 변경되지 않은 상태로 남는다
**그리고** 사용자가 BMAD 작업을 포기하도록 강제하지 않고 워크플로우가 계속된다.

### 스토리 2.4: Git 충돌 및 실행 실패 감지 및 보고

워크플로우 사용자로서,
Git 충돌과 실행 실패가 명확하게 감지되고 보고되기를 원한다.
이를 통해 무엇이 잘못되었고 무엇에 주의가 필요한지 이해할 수 있다.

**인수 조건:**

**주어진 조건** 계획되거나 승인된 Git 작업이 브랜치 충돌, 커밋 실패, 푸시 거부 또는 저장소 상태 불일치를 만난 경우
**행동 시점** 플러그인이 작업을 평가하거나 실행할 때
**기대 결과** 실패 조건을 감지하고 일관되게 분류한다
**그리고** 사용자에게 실패 원인에 대한 명확한 설명을 보고한다.

**주어진 조건** 감사 로깅이 활성화된 경우
**행동 시점** 실패가 감지될 때
**기대 결과** 플러그인은 실패한 작업 결과를 추적 가능한 방식으로 기록한다
**그리고** 로깅 실패 자체가 사용자 워크플로우를 중단하지 않는다.

### 스토리 2.5: 워크플로우를 실패시키지 않는 복구 경로 제공

워크플로우 사용자로서,
자동화가 실패하거나 차단될 때 복구 선택지를 원한다.
이를 통해 워크플로우 진행 상태를 잃지 않고 재시도, 건너뛰기 또는 수동으로 문제를 해결할 수 있다.

**인수 조건:**

**주어진 조건** Git 작업이 거부되거나, 건너뛰어지거나, 실행 중 실패한 경우
**행동 시점** 플러그인이 작업 결과를 처리할 때
**기대 결과** 재시도, 자동화 없이 계속, 수동 해결 후 계속과 같은 복구 경로를 제공한다
**그리고** 사용자가 즉시 조치할 수 있는 방식으로 옵션이 설명된다.

**주어진 조건** 복구 가능한 자동화 실패가 발생한 경우
**행동 시점** 실패 후 워크플로우가 진행될 때
**기대 결과** BMAD 워크플로우가 즉각적인 하드 실패로 처리되지 않는다
**그리고** 사용자가 선택한 복구 경로에 따라 후속 워크플로우 단계가 계속될 수 있다.

## 에픽 3: 완료 및 추적 가능한 전달

사용자가 커밋 및 선택적 푸시 동작으로 워크플로우를 완료하면서 코드, 문서, 기획 아티팩트에 대한 완전한 Git 기반 추적성을 유지할 수 있다.

### 스토리 3.1: 완료 가능한 워크플로우 출력 감지

워크플로우 사용자로서,
워크플로우가 완료 가능한 아티팩트를 생성했는지 플러그인이 판단하기를 원한다.
이를 통해 기록할 의미 있는 출력이 있을 때만 워크플로우 종료 Git 작업이 제안된다.

**인수 조건:**

**주어진 조건** 워크플로우가 완료 단계에 도달한 경우
**행동 시점** 플러그인이 워크플로우 결과를 평가할 때
**기대 결과** 변경된 코드, 문서 또는 기획 아티팩트가 완료 대상인지 판단한다
**그리고** 결정은 워크플로우별 완료 정책을 따른다.

**주어진 조건** 대상 아티팩트가 생성되지 않은 경우
**행동 시점** 완료가 평가될 때
**기대 결과** 플러그인은 불필요한 커밋 작업 제안을 방지한다
**그리고** 비차단 워크플로우 완료 경로를 유지한다.

### 스토리 3.2: 워크플로우 완료 커밋 준비 및 실행

워크플로우 사용자로서,
플러그인이 적절한 시점에 워크플로우 아티팩트로부터 커밋을 준비하기를 원한다.
이를 통해 완료된 작업이 수동 기록 없이 Git 이력에 캡처된다.

**인수 조건:**

**주어진 조건** 완료 가능한 워크플로우 아티팩트가 존재하고 정책에 의해 커밋 동작이 허용된 경우
**행동 시점** 플러그인이 완료를 준비할 때
**기대 결과** 워크플로우의 변경된 아티팩트로 범위가 제한된 커밋 제안을 생성한다
**그리고** 커밋 작업은 앞서 정의된 승인 모델의 적용을 받는다.

**주어진 조건** 사용자가 커밋 작업을 승인한 경우
**행동 시점** 커밋 실행이 수행될 때
**기대 결과** 결과 Git 이력에 관련 코드, 기술 문서 또는 기획 아티팩트가 포함된다
**그리고** 커밋 실패 시 워크플로우 컨텍스트나 추적성 메타데이터를 폐기하지 않고 보고된다.

### 스토리 3.3: 원격 저장소가 사용 가능한 경우에만 푸시 제안

워크플로우 사용자로서,
푸시 동작이 유효하고 관련된 경우에만 제안되기를 원한다.
이를 통해 원격 발행이 방해가 아닌 도움이 된다.

**인수 조건:**

**주어진 조건** 워크플로우 커밋이 성공적으로 완료된 경우
**행동 시점** 플러그인이 푸시 동작을 평가할 때
**기대 결과** 푸시를 제안하기 전에 원격 저장소가 구성되어 있는지 확인한다
**그리고** 유효한 원격이 없으면 푸시 제안을 억제한다.

**주어진 조건** 원격 저장소가 구성되어 있고 푸시 정책이 허용하는 경우
**행동 시점** 푸시 완료가 평가될 때
**기대 결과** 플러그인은 별도의 승인 관리 작업으로 푸시 제안을 생성한다
**그리고** 거부되거나 실패한 푸시가 이미 기록된 로컬 커밋을 무효화하지 않는다.

### 스토리 3.4: 감사를 위한 승인 결과 및 실행 결과 기록

리뷰어 또는 운영자로서,
승인 결과와 Git 실행 결과가 감사 기록에 캡처되기를 원한다.
이를 통해 워크플로우 아티팩트가 어떻게 완료되었는지 이해할 수 있다.

**인수 조건:**

**주어진 조건** Git 작업이 제안, 승인, 거부, 건너뛰기 또는 실행된 경우
**행동 시점** 플러그인이 감사 정보를 기록할 때
**기대 결과** 작업 유형, 결과, 워크플로우 컨텍스트, 타임스탬프를 포함하는 추적 가능한 이벤트를 저장한다
**그리고** 기록은 최소 데이터 로깅 제약을 준수한다.

**주어진 조건** 감사 출력이 클라이언트 로그, 파일 또는 선택적 HTTP 포워딩으로 전달되는 경우
**행동 시점** 감사 기록이 발행될 때
**기대 결과** 플러그인은 구성된 싱크를 최선 노력 방식으로 사용한다
**그리고** 싱크 실패가 워크플로우 완료를 차단하지 않는다.

### 스토리 3.5: 표준 Git 이력을 통한 리뷰어 추적성 유지

리뷰어로서,
워크플로우에서 생성된 아티팩트가 일반 Git 이력 도구로 검사 가능하기를 원한다.
이를 통해 별도의 리뷰 시스템 없이 작성자와 변경 사항을 추적할 수 있다.

**인수 조건:**

**주어진 조건** 워크플로우 출력이 커밋된 경우
**행동 시점** 리뷰어가 표준 Git 이력 도구로 저장소를 검사할 때
**기대 결과** 결과 코드, 기술 문서, 기획 아티팩트가 일반 커밋 이력에서 확인 가능하다
**그리고** 독점적 메타데이터 요구사항 없이 작성자와 변경 책임을 추적할 수 있다.

**주어진 조건** 워크플로우가 코드와 비코드 아티팩트를 모두 생성한 경우
**행동 시점** 완료 커밋이 생성될 때
**기대 결과** 해당 아티팩트가 함께 또는 워크플로우 출력에 맞게 명확히 귀속 가능한 방식으로 기록된다
**그리고** 플러그인은 제품 요구사항에 명시된 감사 가능성 목표를 유지한다.

## 에픽 4: 정책 관리 및 호환성 운영

관리자가 정책 구성을 관리하고, 레거시 호환성을 유지하며, 신뢰할 수 있는 패키징, 릴리스, 런타임 호환성 제어를 통해 플러그인을 운영할 수 있다.

### 스토리 4.1: 브랜치 및 워크플로우 정책 구성 정의 및 정규화

팀 관리자로서,
브랜치 규칙과 워크플로우 정책이 구성 파일에서 정의되기를 원한다.
이를 통해 플러그인 소스 코드를 수정하지 않고 자동화 동작을 변경할 수 있다.

**인수 조건:**

**주어진 조건** 팀이 전역 또는 프로젝트 수준 구성 파일을 제공한 경우
**행동 시점** 플러그인이 구성을 로드하고 정규화할 때
**기대 결과** 브랜치 명명 패턴, 명령 유형 매핑, 병합 대상, 워크플로우 정책이 일관된 유효 형식으로 사용 가능하다
**그리고** 유효하지 않거나 누락된 선택적 값은 지원되는 안전한 기본값으로 폴백한다.

**주어진 조건** 워크플로우 정책 구성이 시간이 지남에 따라 변경되는 경우
**행동 시점** 플러그인이 유효 구성을 해석할 때
**기대 결과** 활성 정책은 최신 적용 가능한 프로젝트 또는 전역 설정을 반영한다
**그리고** 결과 동작은 실행 간에 결정론적으로 유지된다.

### 스토리 4.2: 레거시 구성 호환성 및 브릿지 파일 유지

기존 플러그인 사용자로서,
이전 구성 위치와 형식이 계속 사용 가능하기를 원한다.
이를 통해 파괴적인 마이그레이션 작업 없이 리팩토링된 플러그인 구조를 채택할 수 있다.

**인수 조건:**

**주어진 조건** 레거시 프로젝트 구성 파일만 존재하는 경우
**행동 시점** 플러그인이 시작될 때
**기대 결과** 레거시 구성을 성공적으로 읽고 동등한 런타임 동작을 유지한다
**그리고** 호환성 규칙이 요구하는 경우에만 호환성 브릿지 파일을 생성한다.

**주어진 조건** 현대 및 레거시 구성 파일이 모두 존재하는 경우
**행동 시점** 유효 구성이 해석될 때
**기대 결과** 우선순위 순서가 명시적이고 예측 가능하게 유지된다
**그리고** 호환성 지원이 최신 프로젝트 의도 설정을 조용히 재정의하지 않는다.

### 스토리 4.3: 래퍼를 통한 기존 BMAD 명령 호환성 유지

워크플로우 운영자로서,
리팩토링된 플러그인 래퍼가 기존 BMAD 명령 세트와 동작적으로 호환되기를 원한다.
이를 통해 팀이 사용자 재교육이나 명령 습관 변경 없이 새 구조를 채택할 수 있다.

**인수 조건:**

**주어진 조건** 기존 BMAD 워크플로우 명령이 런타임을 통해 호출된 경우
**행동 시점** 리팩토링된 래퍼가 레거시 코어에 위임할 때
**기대 결과** 명령 이름, 훅 진입점, 코어 동작이 이전 플러그인 계약과 호환되게 유지된다
**그리고** 회귀 검사가 주요 가드 흐름에 대해 래퍼 동작을 레거시 동작과 비교할 수 있다.

**주어진 조건** 래퍼 구조에 플레이스홀더 훅이나 확장 포인트가 존재하는 경우
**행동 시점** 호환성이 평가될 때
**기대 결과** 지원되지 않거나 TODO 경로가 지원되는 기능을 잘못 표시하지 않도록 명시적으로 제한된다
**그리고** 향후 구현 작업이 기존 계약을 깨지 않고 확장할 수 있다.

### 스토리 4.4: 릴리스 아티팩트의 안정적인 빌드 및 패키징

플러그인 유지관리자로서,
빌드 및 릴리스 스크립트가 완전하고 검증 가능한 배포 아티팩트를 생성하기를 원한다.
이를 통해 팀이 지원되는 환경에서 일관되게 플러그인을 설치할 수 있다.

**인수 조건:**

**주어진 조건** 유지관리자가 빌드 및 릴리스 워크플로우를 실행한 경우
**행동 시점** 패키징이 성공적으로 완료될 때
**기대 결과** 릴리스 출력에 번들된 플러그인, 설치기, 구성 템플릿, 매니페스트, 체크섬이 포함된다
**그리고** 버전별 및 `latest` 릴리스 디렉토리가 모두 일관되게 생성된다.

**주어진 조건** 플러그인이 배포용으로 빌드된 경우
**행동 시점** 번들된 아티팩트가 생성될 때
**기대 결과** 지원되는 Node.js 런타임을 대상으로 하고 예상 플러그인 진입 동작을 유지한다
**그리고** 릴리스 메타데이터가 현재 패키지 버전과 아티팩트 해시를 반영한다.

### 스토리 4.5: 래퍼 및 배포 동작에 대한 회귀 커버리지 유지

유지관리자로서,
자동화된 회귀 검사가 래퍼 호환성과 빌드된 아티팩트 동작을 검증하기를 원한다.
이를 통해 리팩토링과 릴리스가 가드된 워크플로우 동작을 조용히 깨지 않는다.

**인수 조건:**

**주어진 조건** 소스, 래퍼, 빌드된 플러그인 아티팩트가 저장소에서 사용 가능한 경우
**행동 시점** 회귀 테스트가 실행될 때
**기대 결과** 레거시, 래퍼, 빌드 변형 간의 주요 명령 프롬프트와 변경 도구 보호를 비교한다
**그리고** 호환성 실패가 릴리스 출력을 신뢰하기 전에 보고된다.

**주어진 조건** 향후 변경이 부트스트랩, 훅 또는 완료 동작에 영향을 미치는 경우
**행동 시점** 테스트가 업데이트되거나 확장될 때
**기대 결과** 회귀 테스트 스위트가 동작 동등성의 품질 게이트로 유지된다
**그리고** 유지관리자가 새 버전 출시 전에 호환성 드리프트를 감지할 수 있다.
