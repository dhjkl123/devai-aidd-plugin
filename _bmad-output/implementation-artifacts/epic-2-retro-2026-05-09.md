## What Went Well

1. **Recovery state machine landed in one shot, semantically.** `src/services/approval/recovery-state.js` defined the canonical states / transitions / blocking scopes / choices, and the orchestrator validated every transition. Even after three review rounds, the state vocabulary itself never needed to change.
2. **Story 2.4's failure envelope `{ ok, status, action, code, message, details, audit, next }` became the single source of truth for failure codes.** Story 2.5 consumed it without re-deriving anything — exactly the boundary the dev notes asked for.
3. **`buildEventEnvelope` + gate-fallback attribution.** The round-2 HIGH fix (attribute via gate.workflow / gate.command when params are null) made every recovery event audit-consumer-friendly without forcing the orchestrator to re-thread workflow/command through every helper.
4. **Shared `buildHookBlockedEvent` envelope helper.** Round-2 MEDIUM. Hook-emitted and orchestrator-emitted `git.action.recovery.blocked` now share canonical `details` keys. Audit consumers no longer branch on emission origin.
5. **Defense-in-depth release signals.** Round-2 LOW: `isActionBlockedByGate` honours `continuationPhase === "terminal"`. Round-3 LOW: `_openBlockedGate` history truthful. Both prevent future race / misleading state, and both came from real review pressure.
6. **Vocabulary disjointness as enforced invariant.** Round-3 LOW: lifted `OUTCOME_ALIASES` and `RECOVERY_CHOICE_ALIASES` into `permission-asked-aliases.js` and added `verifyPermissionAskedAliasDisjointness`. The recovery-first routing precedence is now safe by regression, not by reviewer memory.

## Challenges (Without Blame)

1. **Story 2.5 round-1 HIGH: persistent gate after retry.** The intuition that "the gate is `planned` after retry" turned out to be wrong from the planning hook's perspective — the next pass would see a stale gate and emit `recovery.blocked` instead of republishing approval. Lesson: when a state-machine resets, the *store* must reset too, not just the in-memory representation.
2. **Story 2.5 round-1 MEDIUM: data-shape complete ≠ AC complete.** Recovery options were modeled, the gate carried instructions, the audit was emitted — but no prompt-delivery adapter actually showed the user the choices. AC1 said "options are explained in a way the user can act on immediately"; without delivery, that AC was only data-shape complete.
3. **Story 2.5 round-2 HIGH: attribution loss.** `selected` / `completed` / `blocked` events emitted via `permission-asked` recovery routing carried `workflow: null, command: null` because the gate object didn't persist them. The `offered` event was attributed correctly because the orchestrator received params directly. Lesson: any event the system emits *via state lookup* must source attribution from the state, not from the calling context.
4. **Documentation drift across review rounds.** architecture.md missed Story 2.5's recovery events (caught round-2). sprint-status.yaml never appeared in File List even though every review round touched it (caught round-3). Both fixed.
5. **Task #5 false claim about tool-execute hooks.** Story 2.5 Task #5 read "Extend `permission-asked.js`, `tool-execute-before.js`, `tool-execute-after.js`..." but only `permission-asked.js` was actually modified. The defensible architectural decision (recovery is a planning-time + approval-resolution-time concern) was correct; the task description didn't reflect it. Caught round-3 and amended with a "Round 3 Architectural Note" in Dev Notes.

## Lessons Learned

- **Round-2 and round-3 reviews aren't pedantic — they catch real load-bearing defects.** Each round-N+1 found at least one issue that round-N's reviewer missed. The defects round-2 caught (attribution loss, divergent envelope) were silent in round-1 because the round-1 fix focused on the symptom (gate state) rather than on attribution flow.
- **Audit trail must outlive state cleanup.** When `clearGate` runs, the `selected` + `completed` events have already been emitted. That separation is what made the round-1 HIGH fix safe — clearing the gate didn't lose the audit.
- **Pure prompt builders pay off.** `build-recovery-prompt.js` (round-1 MEDIUM fix) is a pure builder. The orchestrator owns state mutation; the prompt builder only renders. That separation made round-2's `requestRecoveryDecision` adapter a one-liner.
- **Cross-emission consistency requires a shared envelope helper.** When hook emits the same event name as the orchestrator, they must agree on `details` keys. Round-2 MEDIUM proved this with a real audit-consumer-confusing asymmetry.
- **"Hooks thin" is enforceable in practice.** `command-execute-before.js` consults `readRecoveryGate` + `isActionBlockedByGate` and routes through `buildHookBlockedEvent`. It does not own gate logic. `permission-asked.js` routes recovery choices and approval outcomes but delegates state mutation to `consumeApprovalOutcome` / `selectRecoveryChoice`. Tool-execute hooks remain phase-only. The architecture's "hooks thin" rule held under load.
- **Document architectural NO-decisions.** "We did not modify tool-execute hooks because recovery is a planning-time and approval-resolution-time concern" is a real architectural decision. Round-3 codified it in a "Round 3 Architectural Note." Future-self / future-onboarding-engineer thanks current-self.

### Significant Discoveries Affecting Epic 3

> Charlie (Senior Dev): "Two things from Epic 2 directly inform Epic 3."

1. **Recovery for `commit` scopes `WORKFLOW_FINALIZATION` blocking.** This means an unresolved commit recovery gate suppresses Epic 3's push planning automatically. Story 3.3 shouldn't need to re-implement this gate; it just needs to honour `isActionBlockedByGate(gate, "push")` returning blocked when the gate is `commit`-scoped.

2. **Push uses `git-only` blocking scope.** A failed push doesn't invalidate a successful local commit. Story 3.5's reviewer traceability must reflect that — local-only completion is a legitimate terminal state for Epic 3, not a failure.

> Alice (Product Owner): "From the product side, Epic 3 inherits the user-facing recovery vocabulary (retry / continue-without-automation / manual-resolution / abandon). Epic 3 stories should not introduce new vocabulary; they should reuse this one."
